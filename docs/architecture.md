## Design Principles

### 1. Agents are prompts, not code

Each agent is a single markdown file with instructions. No Python, no framework, no runtime. The AI tool reads the markdown and follows the instructions. This means:

- Agents work with any LLM that reads project files
- No dependencies beyond the AI tool itself
- Updating an agent = editing a markdown file

### 2. Canonical source of truth

Each rule has one canonical home, with short summaries allowed elsewhere:

| Rule type | Canonical location |
|-----------|-------------------|
| Workflow, shared rules, quality tier | `CLAUDE.md` |
| Agent-specific steps | `agents/<name>.agent.md` |
| Development standards | `knowledge-base/<topic>.md` |
| Language conventions | `knowledge-base/languages/<lang>.md` |
| Commit conventions | `skills/git/SKILL.md` |

Agents reference rules with a `file.md` plus `§ Section` citation and keep duplicated explanations brief so summaries do not replace the canonical source.

### 3. Deploy, don't link

The CLI copies files into your project rather than symlinking or referencing external paths. This means:

- Your project is self-contained: no dependency on the ai-playbook repo at runtime
- Different projects can run different versions
- Offline-friendly: no network calls needed

### 4. File-based resumability

Agents use markdown artifacts where durable context matters:

| Directory | Created by | Used by |
|-----------|------------|---------|
| `stories/` | story-refiner | all agents |
| `research/` | story-refiner | slice-planner, xp-pair-programmer |
| `plans/` | slice-planner | xp-pair-programmer |
| `audits/` | code-inspector | xp-pair-programmer (to fix findings) |
| `reviews/` | diff-reviewer (when local review output is requested) | humans, release-captain handoff |
| `incidents/` | incident-responder | release-captain, follow-up stories |

These are plain markdown: not stored in `~/.claude/` or any tool-specific location. Any AI tool that reads project files can resume from them.

### 5. Quality tier as a master switch

`CLAUDE.md` declares the default `quality-tier: prototype` or `quality-tier: production`. `.ai-playbook.toml` can override individual agents under `[quality_tiers.agents]`. Every agent has a tier-aware ceremony table that adjusts:

- Number of questions asked
- Depth of review
- Approval gates (block vs save-and-continue)
- Checks skipped or enforced

The root tier cascades through all eight agents unless an agent-specific override is present.

### 6. Model tiers as an adapter contract

Each agent declares a `model: advisor` or `model: executor` field in its frontmatter. These are abstract tier names: not model IDs. The intended tier mapping is declared in `.ai-playbook.toml`. For Claude deploys, `ai-playbook deploy` materializes that mapping into the deployed agent frontmatter (tier name to the configured model) when the value is one Claude Code recognizes, so Claude Code routes each agent natively; for other tools, or for model identifiers Claude Code does not recognize, the actual model selection lives in the AI tool config. Detail: `knowledge-base/model-tier.md` § Deploying to Claude Code.

| Tier       | Role                                  | Common pairings                                           |
|------------|---------------------------------------|-----------------------------------------------------------|
| `advisor`  | Deeper reasoning, planning, review    | Claude Opus, GPT-4, Gemini Pro, Llama 70B                 |
| `executor` | High-volume execution (TDD, drafting) | Claude Sonnet/Haiku, GPT-4 mini, Gemini Flash, Mistral 7B |

This means:

- Swapping models is a config change, not a repo edit
- Anthropic, OpenAI, Google, local-only, and mixed setups all use the same agents
- Single-model setups (e.g. free tier): escalation triggers route to a human review checkpoint instead of a model switch: this preserves strictness

The tier split is a **cost / latency optimisation**, not a quality safety net. TDD red-green, quality gates, the Definition of Done, and the Cognitive Health Gates enforce quality: none of which depend on which model maps to which tier.

Detail: `knowledge-base/model-tier.md`.

## Agent Handoff Workflow

```text
               ┌─────────────┐
  Idea/ticket  │ story-refiner │  Refines, researches, writes story
               └──────┬──────┘
                      │ story file
               ┌──────▼────────┐
               │ slice-planner │  Designs approach, vertical slices, TDD steps
               └──────┬────────┘
                      │ plan file
               ┌──────▼──────┐
               │   xp-pair-programmer   │  Acceptance tests from AC (outer) + unit TDD (inner)
               └──────┬──────┘
                      │ tested code at commit gate
               ┌──────▼────────┐
               │ diff-reviewer │  Verifies acceptance criteria, knowledge base, DoD
               └──────┬────────┘
                      │ approved review
               ┌──────▼──────────┐
               │ release-captain │  Open PR/MR, watch CI, merge on approval, tag, smoke
               └─────────────────┘

  On production incident:  incident-responder  →  triage, postmortem, follow-up artifacts
                                                  (read-only on production; humans
                                                   + release-captain execute mitigations)

  Standalone:  code-inspector (health check)  |  docs-maintainer (documentation, ADRs)
```

Each agent reads artifacts from the previous stage. This is a human- or tool-invoked handoff sequence, not an automated multi-agent orchestrator: the CLI deploys markdown instructions and artifacts, and approval gates stay attached to each step. Code changes move through git status, staged diffs, PRs, and explicit approval gates; durable context moves through markdown artifacts.

**Two-layer testing: acceptance tests + unit TDD:** xp-pair-programmer enforces both acceptance tests and unit TDD on the adopter's code. This follows [Acceptance Test-Driven Development (ATDD)](https://www.agilealliance.org/glossary/atdd/): also known as double-loop TDD: from Freeman & Pryce's *Growing Object-Oriented Software, Guided by Tests*.

1. **Outer loop (acceptance tests)**: for each acceptance criterion (AC) in the story, xp-pair-programmer writes a failing acceptance test (`test_ac_<what>_<condition>`) at the system boundary before any production code. The boundary is architecture-aware: HTTP response for APIs, handler return for Lambda, DOM render for React, CloudFormation assertions for AWS CDK.
2. **Inner loop (unit TDD)**: unit tests drive the implementation one at a time (RED → GREEN → REFACTOR). The feature completes when all acceptance tests pass.

This gives the adopter two independent verification layers: acceptance tests verify the team built the right thing (from the story's AC), unit tests verify the team built it right (from the implementation).

> **Note on terminology.** The "two-layer testing" framing here describes the **double-loop methodology** (outer AT + inner unit TDD). It is distinct from: and complementary to: the **3-layer dev pyramid** prescribed in `knowledge-base/testing.md` § Test Types Quick Guide (Unit / Acceptance / Integration-with-external-services), where E2E moves out into post-deploy. The double-loop is *how you write tests*; the pyramid is *what proportion of each kind*.

## Knowledge Base Loading

Agents don't load the entire knowledge base upfront. Each agent loads the smallest source that can change the task outcome:

1. Use `CLAUDE.md` for the shared rules already in context.
2. Search `knowledge-base/INDEX.md` for the exact topic.
3. Load one KB file or section at a time.
4. Stop once the rule is actionable.

Examples:

- `xp-pair-programmer` loads `testing.md` when writing tests, `security.md` only when touching auth or PII code
- `diff-reviewer` loads `testing.md § Test Quality Rules` instead of the whole testing guide when checking assertions
- `code-inspector` loads files matching the current priority group (P0–P6), not the whole KB

The `knowledge-base/INDEX.md` file is the routing table: it maps topics to files so agents know where to look.

This is a context-efficiency rule, not a lower-quality mode. TDD, security checks, quality gates, and review standards still apply.

## CLI Architecture

The CLI lives in `src/deploy_ai_playbook/`, split across `cli.py`, `paths.py`, `config.py`, `fs.py`, `discovery.py`, `mcp.py`, and `backup.py`. It is a [Typer](https://typer.tiangolo.com/) app with the following commands:

| Command | What it does |
|---------|-------------|
| `init` | Scaffolds the artifact directories and a starter `.ai-playbook.toml`; idempotent |
| `deploy` | Copies agents, KB, skills, templates, and rules into a target project |
| `list` | Shows available agents in the playbook source |
| `status` | Shows deployed agents and their state (active/off) |
| `diff` | Compares source against deployed copy |
| `doctor` | Health check: stale files, missing agents, missing dirs, fingerprint drift |
| `upgrade-check` | CI-friendly drift check (exit 0 = up to date, 1 = drift, 2 = never deployed) |
| `enable` / `disable` | Toggles agents without deleting them |
| `rollback` | Restores previous deployment from backup |
| `config validate` | Validates `.ai-playbook.toml`, pack paths, pack metadata, and tier overrides |
| `artifacts` / `artifact-policy` | Lists story/plan/research/audit/review/incident files; toggles per-project tracking policy |
| `telemetry` | Manages the Claude Stop hook that appends per-session telemetry to `.claude/usage.jsonl` |

The CLI packages the playbook markdown assets inside the wheel via `hatch.build.targets.wheel.force-include`, so `ai-playbook deploy` works from an installed package: no need to clone the repo. The canonical maintenance source remains this repository; installed wheels carry a distribution copy of those assets.

## Harness Engineering

The playbook uses three layers of mechanical enforcement to prevent violations: whether introduced by a human or an AI agent:

```text
┌─────────────────────────────────────────────────┐
│  Context   : docs, rules, knowledge base       │  Agent reads before acting
├─────────────────────────────────────────────────┤
│  Constraints: architecture tests, dep rules    │  Build fails on violation
├─────────────────────────────────────────────────┤
│  Linters   : Make targets + hooks + CI         │  Commit/merge blocked
└─────────────────────────────────────────────────┘
```

### Architecture Tests (`tests/unit/test_architecture.py`)

Enforces the module dependency direction:

```text
paths/config (foundation) → fs/discovery/mcp (middle) → backup (middle) → cli (top)
```

Every module has a declared set of allowed imports. If someone adds a new module, it must declare its rules: or the test fails. The test also detects circular dependencies.

### Quality Gate Contract (`Makefile`)

Adopter projects expose stable targets regardless of language:

```text
make format
make format-check
make lint
make typecheck
make test
make quality
```

Local hook frameworks (pre-commit, lefthook, husky) and CI call these targets or equivalent project scripts. This keeps the agent contract language-neutral while allowing each team to choose its harness.

### CI (`.github/workflows/ci.yml`)

Runs the Make quality contract and universal pre-commit checks remotely on every push/PR. This catches format, lint, tests, config hygiene, and secret-scan violations even if someone skips local hooks. The commit-message hook remains local because CI cannot reconstruct the developer's original commit prompt reliably.

### Cognitive Health Gates

Two human-verification gates prevent AI output from outrunning understanding:

- **Teach-back**: developer explains what the code does, where to debug it, and how it would change (before commit)
- **Commit rationale**: commit body explains *why* the developer chose this approach, with the durable one-line summary in the `Teach-back:` trailer

Detail: `knowledge-base/philosophy.md` § Cognitive Health.

## Testing Strategy (this repo's own tests)

Four test categories for the playbook's own CLI: unit tests, acceptance tests, agent contract tests, and the eval harness. CI enforces format, lint, type check (pyright standard), 95% coverage floor, eval structure, and smoke-test deploy. *(Distinct from the 3-layer pyramid the playbook prescribes for adopters: see `knowledge-base/testing.md`.)*

See **[Testing in CONTRIBUTING.md](../CONTRIBUTING.md#testing)** for the full test layers table and commands.

### Why two test directories?

| Directory | Purpose | Committed? |
|-----------|---------|------------|
| `tests/` | Production test suite: unit + acceptance tests for `src/` | Yes |
| `mutants/tests/` | Mutation testing sandbox: generated copies run against mutated `src/` | No (gitignored) |

`mutants/` is a **generated workspace** that `mutmut` creates during mutation testing. It contains mutated copies of source code and test snapshots used to verify that tests catch real behavioural changes (not just structural ones). The `.gitignore` excludes it: never commit it. If tracked by mistake, remove it with `git rm -r --cached mutants/`.

## Key Trade-offs

| Decision | Trade-off |
|----------|-----------|
| Markdown-only agents (no code framework) | Simpler, more portable: but no runtime validation of agent behavior |
| Deploy (copy) over link (symlink) | Self-contained projects: copy-based deploy is inherently drift-prone, mitigated by `ai-playbook doctor` (fingerprint warning) and `ai-playbook upgrade-check` (CI exit code) |
| Quality tier defaults with per-agent override | One-line root change still cascades everywhere; per-agent exceptions add config to review |
| File-based artifacts over database | Works offline, tool-agnostic, and searchable through `ai-playbook artifacts --query`: but no dashboard or relational history |
| Single CLAUDE.md rules file | Everything in one place: but grows with the project |

For full decision rationale, see [docs/adr/](adr/README.md) (that directory records decisions about the playbook itself).

For the practical impact of these trade-offs, see [Known Limitations](limitations.md).

## Mechanism-Backed vs Agent-Honored Rules

Every rule in the playbook is one of two kinds; knowing which is which prevents the "I thought CI checked this" surprise:

- **Mechanism-backed**: a hook, test, or CI step fails when the rule is broken. Examples: the Teach-back trailer (`harness/check-teachback.sh`, commit-msg hook), citation integrity (`CITATION_BUDGET = 0` in `tests/unit/test_kb_integrity.py`), approval-prompt single-sourcing (`tests/acceptance/test_approval_gate_contracts.py`), the `CLAUDE.md` size budget (`tools/check-claude-md-size.py`), and src/ layering (`tests/unit/test_architecture.py`).
- **Agent-honored**: the rule lives in the prompt; no harness can observe a violation. Examples: read-budget self-tracking, one-agent-at-a-time, model-tier escalation triggers, and the workspace-overlay step of quality-tier resolution (`ai-playbook status` resolves the config override and repo default; the overlay is story-scoped at runtime).

Agent-honored rules are not weaker by accident: they are the rules whose violation is only observable inside a conversation. Their defense is eval rubrics (`evals/`) and human review, not CI. When adding a rule, decide its kind explicitly, and if it *can* be a mechanism, make it one: `knowledge-base/philosophy.md` § Code Quality Signals: a fix needed across sessions belongs in the environment, not the prompt.
